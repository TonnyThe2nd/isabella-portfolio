ControleAcessoCancelas

Documentação técnica completa — arquitetura, fluxos, entidades, integração e configuração

📋 Introdução

O ControleAcessoCancelas é um sistema Windows (.NET Framework 4.6.1 + WPF) para gerenciamento do controle de saída de alunos de uma instituição de ensino. Ele integra leitores RFID de cancelas e catracas físicas, um banco de dados SQL Server centralizado, bases Firebird distribuídas por blocos (catracas), um banco Oracle legado (sistema BHS), e um servidor Redis para pub/sub em tempo real.

🏫

Domínio

Controle de saída de alunos por responsáveis autorizados via RFID em cancelas

⚙️

Stack

.NET 4.6.1 / WPF / WCF / Dapper / SQL Server / Firebird / Oracle / Redis

📡

Hardware

Leitores RFID (cancelas de carro + totem pedestre), catracas físicas por bloco

🖥️

Interfaces

Painel TV de saída, admin de catracas, gestor de responsáveis, web service REST-like

🏗️ Arquitetura do Sistema

O sistema segue uma arquitetura em camadas clássica (UI → BLL → DAL → DB) com múltiplos processos independentes (Windows Services) responsáveis por capturar eventos de hardware e sincronizar bases de dados distintas.

🚦 Fluxo de Saída de Alunos

Este é o fluxo central do sistema. Descreve o ciclo de vida completo de uma saída de aluno, desde a leitura do cartão do responsável na cancela até a finalização do registro.

╔══════════════════════════════════════════════════════════════════╗ ║ FLUXO COMPLETO — SAÍDA DE ALUNO ║ ╚══════════════════════════════════════════════════════════════════╝ ① LEITURA DO RFID NA CANCELA Responsável aproxima cartão RFID na cancela (carro) ou no totem (pedestre) Hardware registra passagem na view Oracle: BHSPROD.V_LOGACE_BARAO campo idcatraca: 201 = cancela carro | 202 = totem pedestre ② SERVIÇO MonitoraCancelas (tick a cada ~N segundos) Lê último id_frequencia do Oracle Compara com LastFrequencia salvo no .INI local Para cada nova frequência: ├─ Identifica CartaoRFID ├─ Busca alunos vinculados ao responsável (GetAlunosByCartaoRfid) ├─ Verifica se responsável já tem registro na sessão atual (HasResponsavel) │ └─ Evita duplicatas na mesma sessão ├─ Para cada aluno do responsável: │ ├─ Cria registro Cancela com flags: Saida=true, Catraca=false, Finalizado=false │ └─ CancelaBll.CadastrarCancela() → INSERT em cancelas.Cancela (SQL Server) └─ Atualiza LastFrequencia no .INI ③ PAINEL TV (ControleAcessoCancelas) — refresh a cada N segundos LoadAlunosParaSair(): ├─ GetAlunosAguardandoSaida(agora, idSessao) → lista "Para Sair" ├─ Filtra por: Curso, Série, Turma, Horário (manhã/bilíngue/tarde) └─ Exibe nome do aluno na tela (coluna esquerda: aguardando saída) LoadAlunosJaSairam(): ├─ GetAlunosPassaramCatraca(dataHoje, idSessao) → lista "Já Saíram" └─ Exibe alunos que passaram pela catraca (coluna direita) ④ PASSAGEM DO ALUNO NA CATRACA Aluno passa pela catraca física (leitora de digital ou cartão) Serviço MonitoraCatracas detecta passagem: ├─ Identifica aluno por RG ├─ Busca cancela pendente no SQL Server └─ PUT /catraca/aluno/{token}/{idAluno} → UpdateCatracaAluno() └─ CancelaBll.InsertAlunoPassouCatraca() └─ UPDATE Cancela: Catraca=true, Saida=false ⑤ FINALIZAÇÃO AUTOMÁTICA (a cada ~20 segundos) CancelaBll.DeleteAlunoCatraca() └─ UpdateCancelaFinalizados() → marca Finalizado=true para registros antigos CASO ESPECIAL: Saída Manual (operador) InserirSaidaManualAluno(idAluno, idSessao) └─ UPDATE Cancela: Finalizado=true, DataHoraSaidaManual=now CASO ESPECIAL: Lista de Espera (Aguardando) Aluno com horário diferente → InsertAguardando() └─ UPDATE Cancela: Aguardando=true Quando chega o horário → PassarDeAguardandoPara(estado) └─ Estado = ParaSair (Saida=true) ou Finalizado
Sessão: Todos os registros de cancela são vinculados a uma Sessao aberta. Sem sessão ativa, o serviço MonitoraCancelas ignora as passagens. A sessão é aberta/fechada manualmente pela interface administrativa.

🔄 Estados da Cancela (Máquina de Estados)

Cada registro na tabela cancelas.Cancela representa um aluno em um determinado estado dentro do processo de saída. As flags Saida, Catraca, Finalizado e Aguardando são mutuamente exclusivas por convenção.

EstadoSaidaCatracaFinalizadoAguardandoSignificado
Na lista de saída true falsefalsefalse Responsável passou na cancela. Aluno aguardando na tela.
Aguardando falsefalsefalse true Aluno em horário diferente — na lista de espera.
Passou na catraca false true falsefalse Aluno passou fisicamente pela catraca. Coluna direita do painel.
Finalizado falsefalse true false Processo concluído. Saída manual ou finalização automática.
TRANSIÇÕES DE ESTADO: [INICIAL: Responsável passa na cancela] │ ▼ Saida=true ──────────────────────────────────────────────────→ Aguardando=true │ (na lista) (horário diferente) │ │ ▼ (aluno passa na catraca) ▼ (horário chega) Catraca=true Saida=true ou Finalizado=true │ ▼ (ciclo automático ~20s) Finalizado=true ←────── Saída manual pelo operador

🔑 Ciclo de Sessão

A Sessão é o elemento central de controle do dia letivo. Todos os registros de cancela, frequência e autorização são vinculados a uma sessão ativa.

OperaçãoMétodo BLLEfeito
Abrir sessãoSessaoBll.AbrirSessao()Insere registro na tabela Sessao com DataHoraAbertura=now
Obter sessão abertaSessaoBll.GetSessaoAberta()Retorna sessão sem DataHoraFechamento. Retorna null se inexistente.
Fechar sessãoSessaoBll.FecharSessao()Seta DataHoraFechamento=now na sessão em aberto
Atenção: Sem sessão aberta, o serviço MonitoraCancelas interrompe o processamento e registra log de aviso. A interface TV também não exibe dados enquanto não há sessão ativa.

⚡ Windows Services

MonitoraCancelas — Detalhe do Tick

Serviço mais crítico do sistema. Lê continuamente o Oracle BHS e insere registros no SQL Server.

// Lógica simplificada do Tick()
public static void Tick() {
    GetSessaoAberta();
    if (SessaoAberta == null) return;   // sem sessão → ignora
    
    LastFrequencia = GetFrequencia();   // última passagem no Oracle
    var iLast = int.Parse(Config.LastFrequencia); // salvo em INI
    
    while (iLast <= LastFrequencia.IdFrequencia) {
        var freq = GetFrequencia(iLast);
        var responsaveis = ResponsavelAlunoBll.GetAlunosByCartaoRfid(freq.CartaoRfid);
        
        if (!CancelaBll.HasResponsavel(sessao, responsaveis[0].IdResponsavel)) {
            // insere um registro por aluno do responsável
            CancelaBll.CadastrarCancela(cancela);
        }
        iLast++;
    }
    Config.SaveFrequencia(); // persiste último ID no .INI
}

MonitoraAlunos — Sincronização de Alunos

Sincroniza diariamente (ou sob demanda) o cadastro de alunos do SQL Server para as bases de catracas, por bloco de acesso.

// Fluxo do Tick()
GetAlunosNoSql()  // lista AlunoCatraca do SQL

🔍 Consultas SQL Críticas

Alunos aguardando saída (GetAlunosAguardandoSaida)

-- Busca alunos cujo responsável já passou na cancela (Saida=1)
-- e que ainda não passaram na catraca nem foram finalizados
SELECT c.*, ra.*, a.*, r.*
FROM cancelas.Cancela c
JOIN cancelas.ResponsavelAluno ra ON c.idResponsavelAluno = ra.idResponsavelAluno
JOIN cancelas.Aluno a ON ra.idAluno = a.idAluno
JOIN cancelas.Responsavel r ON ra.idResponsavel = r.idResponsavel
WHERE c.idSessao = @idSessao
  AND c.saida = 1
  AND c.finalizado = 0

Passagens na cancela (histórico)

SELECT r.Nome, c.idAluno, c.idResponsavelAluno,
       c.DataHoraCancela, c.DataHoraCatraca, c.DataHoraSaidaManual,
       va.Descricao
FROM cancelas.cancela c
JOIN cancelas.ResponsavelAluno ra ON c.idResponsavelAluno = ra.idResponsavelAluno
JOIN cancelas.Responsavel r ON ra.idResponsavel = r.idResponsavel

🏷️ Bloco × Filtro de Cursos

A chave Bloco no App.config não é apenas um rótulo — ela controla quais cursos do range configurado em Cursos serão efetivamente aceitos pelo método Configuracao.GetCursos(). O range no config é o universo possível; o Bloco é o filtro que recorta esse universo.

Atenção: Apenas Bloco1 e Infantil possuem filtro de curso implementado. Os demais valores de Bloco (Bloco2, Bloco3, ExternoBloco1) não entram em nenhum if dentro do loop de ranges — o que significa que cursos especificados como range (ex: 124-126) são silenciosamente ignorados para esses blocos. Apenas IDs avulsos (sem hífen) passam normalmente.

Lógica do GetCursos()

public static List<int> GetCursos()
{
    var cursos = new List<int>();
    foreach (var cursosSet in Cursos.Split(','))    // ex: "124-126" ou "150-165"
    {
        var set = cursosSet.Split('-');
        if (set.Length == 2)    // é um range
        {
            for (int i = set_i; i <= set_f; i++)
            {
                if (IsBloco1)
                {
                    if (124 <= i && i <= 126) cursos.Add(i);  // hardcoded!
                }
                else if (IsInfantil)
                {
                    if (150 <= i && i <= 165) cursos.Add(i);  // hardcoded!
                }
                // Bloco2, Bloco3, ExternoBloco1 → não entram em nenhum if → curso ignorado
            }
        }
        if (int.TryParse(cursosSet, out var curso_id))   // é um ID avulso (sem hífen)
        {
            cursos.Add(curso_id);   // adicionado sem filtro de Bloco
        }
    }
    return cursos;
}

Comportamento por Bloco com o config padrão (Cursos=124-126,150-165)

Bloco (App.config)Cursos efetivos geradosObservação
Bloco1 [124, 125, 126] Apenas o range 124–126 passa. O range 150–165 é percorrido mas nenhum if aceita esses valores para Bloco1 → descartados.
Infantil [150, 151, ..., 165] Apenas o range 150–165 passa. O range 124–126 é percorrido mas descartado, pois não cai no if (IsInfantil).
Bloco2 / Bloco3 / ExternoBloco1 [] (lista vazia para ranges) Nenhum if cobre esses blocos dentro do loop de range. IDs avulsos (sem hífen) ainda seriam adicionados, mas o config padrão usa apenas ranges.
Por que os ranges estão hardcoded? Os valores 124–126 e 150–165 são os códigos de curso do ERP da instituição para, respectivamente, o Ensino Fundamental (Bloco1) e a Educação Infantil. A chave Cursos no config define o universo a percorrer, mas a decisão de quais IDs são válidos para cada Bloco está embutida no código — o que significa que adicionar um novo bloco requer alteração no código-fonte do GetCursos(), não apenas no config.

📄 Arquivos INI

O cliente WPF ControleAcessoCancelas usa um arquivo WebServices.ini para configurar as URLs dos serviços. Lido pela classe IniFile / WsAddress.

[BASE]
URLBase=http://servidor:porta/

[SESSAO]
URLSessaoSVC=sessao.svc/

[SAIDA]
URLSaidaSVC=saida.svc/

O MonitoraCancelas usa um INI local para persistir o LastFrequencia entre reinicializações do serviço:

[CONFIG]
LastFrequencia=123456
LogType=Verbose
Dica de operação: Em caso de perda de eventos de cancela, basta decrementar o valor de LastFrequencia no INI do MonitoraCancelas e reiniciar o serviço. Ele reprocessará os eventos a partir daquele ID.
